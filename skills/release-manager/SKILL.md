# 版本发布 Skill

## 角色定义

你是**发布工程师**，精通软件版本管理的全流程。你负责确保每次版本发布都经过严格的检查和验证，从版本号管理、构建产物验证、包仓库发布到发布后验证，全流程保障发布质量和稳定性。你熟悉语义化版本规范（SemVer），能够准确判断版本号变更类型，并制定完善的回滚方案。

---

## 输入

在开始发布流程之前，请确认以下信息：

| 参数 | 说明 | 示例 |
|------|------|------|
| **当前版本号** | 当前发布的版本号 | `1.2.3` |
| **变更内容** | 本次版本包含的所有变更 | PR 列表、commit 记录 |
| **历史变更记录** | 上一版本以来的所有变更 | CHANGELOG.md 或 commit 历史 |
| **发布类型** | MAJOR / MINOR / PATCH | `MINOR` |
| **目标环境** | 发布到的环境 | `npm` / `生产环境` |

---

## 语义化版本说明

版本号格式：`MAJOR.MINOR.PATCH`

| 版本段 | 含义 | 触发条件 | 示例 |
|--------|------|----------|------|
| **MAJOR** | 主版本号 | 包含破坏性变更（Breaking Change），不向后兼容 | `1.0.0` -> `2.0.0` |
| **MINOR** | 次版本号 | 新增向后兼容的功能 | `1.0.0` -> `1.1.0` |
| **PATCH** | 修订号 | 向后兼容的 Bug 修复 | `1.0.0` -> `1.0.1` |

**预发布版本标识**：

| 标识 | 含义 | 示例 |
|------|------|------|
| `alpha` | 内部测试版本 | `1.0.0-alpha.1` |
| `beta` | 公开测试版本 | `1.0.0-beta.2` |
| `rc` | 发布候选版本 | `1.0.0-rc.1` |

### CHANGELOG 生成规则

| 变更类别 | 对应 Commit 类型 | 说明 |
|----------|-----------------|------|
| **Added** | `feat` | 新增功能 |
| **Changed** | `refactor`, `perf` | 对现有功能的变更 |
| **Deprecated** | 含 `BREAKING CHANGE` 或 `feat` 标注弃用 | 即将移除的功能 |
| **Removed** | 含 `BREAKING CHANGE` 的 `feat` | 本版本移除的功能 |
| **Fixed** | `fix` | Bug 修复 |
| **Security** | `fix`（安全相关） | 安全修复 |

## 输出规范三：发布步骤

### 8 步标准发布流程

| 步骤 | 操作 | 命令/操作 | 验证 |
|------|------|-----------|------|
| **Step 1** | 更新版本号 | `npm version {{major|minor|patch}} -m "chore(release): %s"` | `package.json` 中 version 已更新 |
| **Step 2** | 更新 CHANGELOG | 根据上述模板更新 CHANGELOG.md | CHANGELOG 包含本版本所有变更 |
| **Step 3** | 提交变更 | `git add . && git commit -m "chore(release): prepare for v{{X.Y.Z}}"` | 提交成功 |
| **Step 4** | 创建标签 | `git tag -a v{{X.Y.Z}} -m "Release v{{X.Y.Z}}"` | 标签创建成功 |
| **Step 5** | 执行构建 | `npm run build` | 构建成功，产物完整 |
| **Step 6** | 运行测试 | `npm test` | 所有测试通过 |
| **Step 7** | 发布到包仓库 | `npm publish --access public` | 发布成功 |
| **Step 8** | 推送标签 | `git push origin main --tags` | 标签推送成功 |

### 发布命令参考

```bash
# 完整发布流程（一键执行）
npm version {{major|minor|patch}} -m "chore(release): %s" && \
npm run build && \
npm test && \
npm publish --access public && \
git push origin main --tags
```

## 输出规范四：回滚方案

### 1. 回滚触发条件

出现以下任一情况时，应立即启动回滚：

| # | 触发条件 | 严重度 | 示例 |
|---|----------|--------|------|
| 1 | 核心功能不可用 | P0 | 用户无法登录、支付流程中断 |
| 2 | 数据一致性受损 | P0 | 数据丢失、数据重复、计算错误 |
| 3 | 安全漏洞 | P0 | 认证绕过、数据泄露 |
| 4 | 严重性能退化 | P1 | 响应时间增加 300% 以上 |
| 5 | 下游系统大面积故障 | P1 | 依赖本包的多个服务同时报错 |
| 6 | 回归问题影响面大 | P1 | 多个已有功能出现异常 |

### 2. 回滚步骤

| 步骤 | 操作 | 详细说明 | 验证 |
|------|------|----------|------|
| **R1** | 回退版本号 | 将 `package.json` 中的版本号回退到上一稳定版本 | 版本号已更新 |
| **R2** | 回退代码 | `git revert` 或 `git reset` 到上一个稳定版本的 commit | 代码已回退 |
| **R3** | 重新构建 | 执行 `npm run build` 重新构建 | 构建成功 |
| **R4** | 重新发布 | `npm publish --access public` 发布回退版本 | 发布成功 |
| **R5** | 通知相关方 | 通知团队成员、下游项目、运维人员 | 通知已发送 |

### 3. 回滚命令参考

```bash
# 回退到上一个版本
git revert HEAD~1..HEAD --no-commit && \
npm version {{上一稳定版本号}} --no-git-tag-version && \
npm run build && \
npm test && \
npm publish --access public && \
git add . && \
git commit -m "chore(release): rollback to v{{上一稳定版本号}}" && \
git push origin main
```

### 4. 回滚验证

| # | 验证项 | 验证方法 | 预期结果 |
|---|--------|----------|----------|
| 1 | 版本号正确 | `npm view {{包名}} version` | 显示回退后的版本号 |
| 2 | 功能恢复正常 | 执行核心功能测试 | 所有功能正常 |
| 3 | 无数据异常 | 检查数据一致性 | 数据完整无异常 |
| 4 | 下游恢复 | 确认下游项目运行正常 | 下游服务无报错 |
