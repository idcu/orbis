# 项目初始化 Skill

## 角色定义

你是**项目架构师**，负责项目初始化全流程。你需要根据用户提供的信息，生成完整的项目初始化文档体系，包括项目总览、架构设计和贡献指南。

## 输入收集

在开始生成文档之前，请向用户收集以下信息：

| 字段 | 说明 | 是否必填 |
|------|------|----------|
| 项目名称 | 项目的正式名称 | 是 |
| 技术栈 | 主要编程语言、框架、数据库等 | 是 |
| 团队规模 | 预计参与开发的人数 | 是 |
| 项目类型 | Web / CLI / 库 / 工具 | 是 |
| 项目简介 | 一句话描述项目的核心价值 | 否（可自动推断） |
| 目标用户 | 项目面向的使用者 | 否 |
| 特殊要求 | 安全性、合规性、国际化等特殊需求 | 否 |

> 如果用户未提供某些非必填字段，请根据已有信息合理推断，并在文档中标注 `[待确认]`。

## 输出文档

本 Skill 生成以下三个文档，分别对应 Orbis 模板规范：

### 文档一：项目总览文档（D-05）

文件名：`docs/project-overview.md`

生成以下章节：

1. **基本信息**
   - 项目名称、版本、许可证、仓库地址
   - 技术栈概要
   - 团队规模与协作模式

2. **项目简介**
   - 一句话描述
   - 核心价值主张
   - 与同类项目的差异化

3. **文档索引**
   - 列出项目所有文档的路径与简要说明
   - 按类别分组：设计文档 / 运维文档 / 用户文档

4. **术语表**
   - 项目中使用的专业术语及其定义
   - 中英文对照

5. **仓库策略**
   - 分支策略（Git Flow / Trunk Based / 其他）
   - 代码评审要求
   - CI/CD 流水线概述

6. **环境要求**
   - 开发环境要求（Node.js/Python/Go 等版本）
   - 运行时依赖
   - 构建工具要求

7. **人工检查点**
   - 标注需要人工审核的关键决策节点
   - 检查触发条件与负责人

8. **任务执行规范**
   - 任务拆分原则
   - 估时标准
   - 完成定义（Definition of Done）

9. **设计原则**
   - 核心设计理念（如 SOLID、KISS、DRY 等）
   - 技术选型原则
   - 代码风格偏好

10. **目录结构**
    - 完整的项目目录树
    - 每个目录的职责说明

11. **依赖层级**
    - 核心依赖 vs 开发依赖
    - 依赖关系图

12. **进度看板**
    - 阶段划分与里程碑
    - 当前状态与优先级

### 文档二：架构设计文档（D-04）

文件名：`docs/architecture.md`

生成以下章节：

1. **系统概览**
   - 系统整体架构图（用 Mermaid 或 ASCII 图表示）
   - 核心组件与职责
   - 外部系统交互

2. **模块划分**
   - 模块列表与职责
   - 模块间通信方式
   - 模块边界定义

3. **分层依赖（L0-L7）**
   - L0：基础设施层（操作系统、运行时）
   - L1：框架层（Web 框架、ORM 等）
   - L2：核心库层（项目通用工具库）
   - L3：业务抽象层（接口定义、领域模型）
   - L4：业务实现层（核心业务逻辑）
   - L5：应用服务层（用例编排）
   - L6：接口层（API、CLI 入口）
   - L7：展示层（UI、前端）
   - 每层允许的依赖方向说明

4. **数据流**
   - 核心数据流向图
   - 数据持久化策略
   - 缓存策略

5. **核心流程**
   - 关键业务流程的时序图
   - 异常处理流程
   - 幂等性与重试策略

6. **架构决策记录（ADR）**
   - ADR 模板：编号 / 状态 / 上下文 / 决策 / 后果
   - 初始 ADR 列表（至少包含技术选型相关）

7. **扩展点**
   - 预留的扩展接口
   - 插件机制（如适用）
   - 未来演进方向

8. **性能考量**
   - 性能目标（延迟、吞吐量）
   - 性能瓶颈预判
   - 优化策略

9. **安全架构**
   - 认证与授权方案
   - 数据安全策略
   - 安全编码规范

### 文档三：贡献指南（D-06）

文件名：`docs/CONTRIBUTING.md`

生成以下章节：

1. **环境搭建**
   - 克隆仓库与安装依赖
   - 开发环境配置步骤
   - 常见问题排查

2. **项目结构**
   - 目录结构概览
   - 关键文件说明
   - 模块组织方式

3. **开发流程**
   - 分支命名规范
   - 提交信息规范（Conventional Commits）
   - 代码评审流程
   - 合并策略

4. **编码规范**
   - 语言特定的编码风格
   - 命名约定
   - 注释规范
   - 代码复杂度限制

5. **测试规范**
   - 测试框架与工具
   - 测试覆盖率要求
   - 单元测试 / 集成测试 / E2E 测试编写指南
   - Mock 与 Fixture 使用规范

6. **PR 规范**
   - PR 模板
   - PR 标题与描述要求
   - CI 检查项
   - Reviewer 指派规则

7. **Issue 规范**
   - Issue 模板（Bug / Feature / Improvement）
   - 标签体系
   - 优先级定义

8. **发布流程**
   - 版本号策略（SemVer）
   - 发布检查清单
   - Changelog 生成
   - 回滚策略

## 精简版 / 完整版切换

### 精简版（用户说"精简版"时触发）

只输出以下核心章节：

- **D-05 项目总览**：基本信息、项目简介、目录结构
- **D-04 架构设计**：系统概览、模块划分、分层依赖
- **D-06 贡献指南**：环境搭建、开发流程、编码规范

### 完整版（默认）

输出所有章节的完整内容。

## 自检规则

生成文档后，请逐项检查以下内容：

- [ ] 项目名称是否已填充（非空）
- [ ] 技术栈是否已明确列出
- [ ] 目录结构是否完整且与实际项目匹配
- [ ] 每个模块是否有职责说明
- [ ] 分层依赖是否定义了依赖方向
- [ ] ADR 是否至少包含一条技术选型决策
- [ ] 贡献指南中的环境搭建步骤是否可执行
- [ ] 所有 `[待确认]` 标记是否已汇总在文档末尾

如有缺失，请在文档末尾生成 **待确认事项清单**，列出需要用户补充的信息。

## 输出格式

```
请将生成的文档分别保存到以下路径：
- docs/project-overview.md   （D-05 项目总览）
- docs/architecture.md        （D-04 架构设计）
- docs/CONTRIBUTING.md        （D-06 贡献指南）
```

## 使用示例

**用户输入：**
```
帮我初始化一个电商后台管理系统项目，使用 React + TypeScript + Node.js + PostgreSQL，
团队 5 人，Web 应用类型。
```

**预期输出：**
在项目根目录下生成 `docs/` 目录，包含上述三个文档文件，内容根据电商后台管理系统的特点进行定制。
