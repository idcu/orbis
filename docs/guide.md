# 📖 模板使用指南

> 本指南帮助你快速了解 Orbis 模板体系的使用方法、模板间的关联关系以及选择建议。

## 快速开始

1. 根据你的场景，从下方选择合适的模板
2. 复制模板内容，将 `｛｛占位符｝｝` 替换为实际信息
3. 按模板结构填写各项内容

## 模板分类总览

| 领域 | 编号 | 模板数 | 适用场景 |
|:---:|:---:|:---:|:---------|
| **A — 模块开发** | A-01 ~ A-09 | 9 | 创建新模块、编写测试、定义接口 |
| **B — 工程化** | B-01 ~ B-09 | 9 | 配置 CI/CD、Git 钩子、文档站 |
| **C — 质量保障** | C-01 ~ C-07 | 7 | 代码评审、验收、PR 检查、性能基准 |
| **D — 项目管理** | D-01 ~ D-06 | 6 | 需求文档、技术方案、架构设计 |
| **E — 开发任务** | E-01 ~ E-06 | 6 | 功能开发、Bug 修复、重构、发布 |

## 精简版 vs 完整版

| 维度 | 精简版 | 完整版 |
|------|--------|--------|
| 适用场景 | 快速起步、小型项目、内部工具 | 正式项目、团队协作、需要全面规范 |
| 章节数量 | 核心要素 | 全量要素 |
| 填写时间 | 5~15 分钟 | 15~60 分钟 |
| 切换方式 | 页面内一键切换（导航栏按钮） | 页面内一键切换 |

**建议**：项目初期使用精简版快速起步，随着项目成长逐步切换到完整版。

## 模板关联关系与使用顺序

### 项目启动阶段

```
D-05 项目总览 → D-01 需求文档 → D-02 技术方案 → D-04 架构设计
```

1. **D-05 项目总览**：创建项目，定义基本信息、文档索引、开发规范
2. **D-01 需求文档**：详细描述功能需求、非功能需求、验收标准
3. **D-02 技术方案**：设计技术实现方案，对比备选方案，评估风险
4. **D-04 架构设计**：设计系统架构、模块划分、分层依赖

### 模块开发阶段

```
A-01 创建检查 → A-02 清单文件 → A-03 构建配置 → A-04 测试配置
→ A-05 测试文件 → A-06 README → A-07 变更日志 → A-09 接口定义
```

1. **A-01**：按检查清单逐项确认新模块的创建规范
2. **A-02 ~ A-04**：配置模块的清单文件、构建和测试
3. **A-05**：编写覆盖 5 种测试类别的测试文件
4. **A-06 ~ A-09**：编写文档、变更日志和接口定义

### 工程化配置阶段

```
B-09 根仓库配置 → B-08 工作区配置 → B-01 CI流水线 → B-05 Git钩子
→ B-02 发布流水线 → B-03 文档部署
```

1. **B-09**：配置根仓库的包清单、代码检查、忽略规则
2. **B-08**：配置工作区、目录协议、版本对齐
3. **B-01**：配置 CI 流水线（lint → test → build → E2E）
4. **B-05**：配置本地 Git 钩子（pre-commit、commit-msg、pre-push）
5. **B-02 ~ B-03**：配置发布流水线和文档部署

### 开发执行阶段

```
E-01 新功能 / E-02 Bug修复 / E-03 重构 / E-04 性能优化
→ C-05 PR检查 → C-01 代码评审 → C-02 评审修复
→ C-03 模块验收 → C-04 发布检查 → E-06 版本发布
```

1. **E-01 ~ E-04**：根据任务类型选择对应的开发模板
2. **C-05**：提交 PR 前按检查清单逐项确认
3. **C-01 → C-02**：代码评审 → 修复评审问题
4. **C-03**：新模块开发完成后进行验收
5. **C-04 → E-06**：发布前检查 → 执行版本发布

### 团队协作

- **D-06 贡献指南**：为外部贡献者提供参与指引
- **B-04 多仓库同步**：在多仓库间同步代码
- **B-06 共享脚本**：统一依赖检查、版本同步、构建、测试、发布

## 与 AI 编程工具配合使用

模板是 AI 编程的"提示词工程基础设施"，以下是配合使用的建议：

1. **直接粘贴**：将模板内容复制到 AI 对话中，填入你的具体信息
2. **结构化输入**：AI 收到结构化的模板后，输出质量会显著提升
3. **检查清单驱动**：将检查清单作为 AI Code Review 的 Prompt
4. **模板即契约**：模板定义了输出应该包含什么、格式是什么、质量标准是什么

## 注意事项

1. 所有模板使用 `｛｛占位符｝｝` 格式（全角花括号），复制使用时替换为实际内容
2. 标记为 `[可选]` 的章节可根据项目实际情况决定是否填写
3. 模板为通用设计，可根据团队实际情况调整和定制
4. 建议将常用模板保存为团队模板，减少重复填写
