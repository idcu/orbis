# B-F02 发布流水线模板（完整版）

> **模板 ID**: B-F02
> **模板名称**: 发布流水线模板（完整版）
> **版本**: 1.0.0
> **适用场景**: 完整发布流程，包含版本管理、包发布、发布说明生成及发布后文档更新

---

## 概述

本模板提供一个完整发布流水线，包含 **4 个作业**：

1. **版本 PR** - 自动创建版本号变更的合并请求
2. **发布** - 构建并发布到包注册表，包含产物签名与来源证明
3. **发布** - 生成变更日志和发布说明，创建发布记录
4. **发布后处理** - 更新文档站点，同步相关仓库

---

## 流水线配置

```yaml
# ｛｛项目名称｝｝ 发布流水线（完整版）
# 适用于 CI 平台

# ====== 全局变量 ======
variables:
  PROJECT_NAME: "｛｛项目名称｝｝"
  REGISTRY_URL: "｛｛包注册表地址｝｝"
  RELEASE_BRANCH: "release/v｛｛版本号｝｝"

# ====== 并发控制 ======
concurrency:
  group: "release-$｛｛分支名称｝｝"
  cancel-in-progress: false  # 发布不允许取消

# ====== 手动触发 ======
triggers:
  manual:
    inputs:
      version-bump:
        description: "版本升级类型"
        type: "choice"
        options: ["patch", "minor", "major"]
        default: "patch"
      dry-run:
        description: "试运行（不实际发布）"
        type: "boolean"
        default: false

# ====== 作业 1: 版本 PR ======
jobs:
  version-pr:
    name: "创建版本 PR"
    runs-on: "｛｛运行环境｝｝"
    if: "$｛｛手动触发｝｝"
    steps:
      - name: 检出代码
        uses: "checkout@v4"
        with:
          token: "$｛｛写权限令牌｝｝"

      - name: 安装依赖
        run: "｛｛安装依赖命令｝｝"

      - name: 确定新版本号
        id: version
        run: |
          CURRENT_VERSION=$(｛｛获取当前版本命令｝｝)
          echo "当前版本: $CURRENT_VERSION"
          # 根据输入确定新版本号
          NEW_VERSION=$(｛｛计算新版本命令｝｝ --bump $｛｛版本升级类型｝｝)
          echo "新版本: $NEW_VERSION"
          echo "new_version=$NEW_VERSION" >> $GITHUB_OUTPUT

      - name: 更新版本号
        run: |
          ｛｛更新版本号命令｝｝ $｛｛步骤输出.新版本号｝｝

      - name: 生成变更日志
        run: |
          ｛｛生成变更日志命令｝｝ --release $｛｛步骤输出.新版本号｝｝

      - name: 提交版本变更
        run: |
          git config user.name "｛｛机器人用户名｝｝"
          git config user.email "｛｛机器人邮箱｝｝"
          git add -A
          git commit -m "chore: 发布版本 v$｛｛步骤输出.新版本号｝｝"
          git push origin "｛｛版本分支名｝｝"

      - name: 创建合并请求
        uses: "create-pr@v4"
        with:
          title: "chore: 发布版本 v$｛｛步骤输出.新版本号｝｝"
          body: |
            ## 版本发布 PR

            - **版本**: v$｛｛步骤输出.新版本号｝｝
            - **类型**: $｛｛版本升级类型｝｝
            - **变更日志**: 见下方 diff

            ---
            请审核后合并以触发发布流程。
          branch: "｛｛版本分支名｝｝"
          base: "｛｛默认分支｝｝"

  # ====== 作业 2: 发布到包注册表 ======
  publish:
    name: "发布到包注册表"
    runs-on: "｛｛运行环境｝｝"
    if: "$｛｛分支名称 == 默认分支｝｝"
    needs: []  # 由版本 PR 合并触发
    steps:
      - name: 检出代码
        uses: "checkout@v4"

      - name: 安装依赖
        run: "｛｛安装依赖命令｝｝"

      - name: 运行测试
        run: "｛｛运行测试命令｝｝"

      - name: 构建
        run: "｛｛构建命令｝｝"

      - name: 产物签名
        run: |
          # 对构建产物进行数字签名
          ｛｛签名命令｝｝ ｛｛构建产物路径｝｝ --key "$｛｛签名密钥｝｝"

      - name: 生成来源证明
        uses: "provenance@v2"
        with:
          builder-id: "$｛｛构建器ID｝｝"

      - name: 发布到 ｛｛包注册表地址｝｝
        run: "｛｛发布命令｝｝"
        env:
          REGISTRY_TOKEN: "$｛｛发布令牌｝｝"

      - name: 验证发布
        run: |
          # 确认包已成功发布
          ｛｛验证发布命令｝｝ $｛｛步骤输出.新版本号｝｝

  # ====== 作业 3: 创建发布记录 ======
  release:
    name: "创建发布记录"
    runs-on: "｛｛运行环境｝｝"
    needs: [publish]
    steps:
      - name: 检出代码
        uses: "checkout@v4"

      - name: 获取版本号
        id: version
        run: |
          VERSION=$(｛｛获取当前版本命令｝｝)
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: 生成变更日志
        run: |
          ｛｛生成变更日志命令｝｝ --release $｛｛步骤输出.版本号｝｝ --output CHANGELOG.md

      - name: 生成发布说明
        id: notes
        run: |
          # 基于变更日志生成发布说明
          ｛｛生成发布说明命令｝｝ > release-notes.md

      - name: 创建 Git 标签
        run: |
          git config user.name "｛｛机器人用户名｝｝"
          git config user.email "｛｛机器人邮箱｝｝"
          git tag -a "v$｛｛步骤输出.版本号｝｝" -m "Release v$｛｛步骤输出.版本号｝｝"
          git push origin "v$｛｛步骤输出.版本号｝｝"

      - name: 创建发布记录
        uses: "create-release@v4"
        with:
          tag: "v$｛｛步骤输出.版本号｝｝"
          name: "v$｛｛步骤输出.版本号｝｝"
          body-file: "release-notes.md"
          artifacts: "｛｛构建产物路径｝｝"

  # ====== 作业 4: 发布后处理 ======
  post-release:
    name: "发布后处理"
    runs-on: "｛｛运行环境｝｝"
    needs: [release]
    steps:
      - name: 检出代码
        uses: "checkout@v4"

      - name: 更新文档
        run: |
          # 更新文档中的版本引用
          ｛｛更新文档版本命令｝｝ $｛｛步骤输出.版本号｝｝

      - name: 触发文档部署
        run: |
          # 通知文档站点重新构建
          ｛｛触发文档部署命令｝｝

      - name: 通知下游仓库
        run: |
          # 通知依赖此包的仓库更新版本
          ｛｛通知下游命令｝｝ --version $｛｛步骤输出.版本号｝｝

      - name: 发布完成通知
        run: |
          # 发送发布完成通知（如聊天工具、邮件等）
          ｛｛发送通知命令｝｝ \
            --title "发布成功: v$｛｛步骤输出.版本号｝｝" \
            --body "版本 v$｛｛步骤输出.版本号｝｝ 已成功发布到 ｛｛包注册表地址｝｝"
```

## 发布流程

```
手动触发 ──→ 版本 PR ──→ 审核合并 ──→ 发布到注册表 ──→ 创建发布记录 ──→ 发布后处理
                                  ↑                    ↓
                              签名 + 来源证明      Git 标签 + 发布说明
```

---

## 适配指南

- **GitHub Actions**: `uses` 对应 marketplace action，`secrets` 通过 `$｛｛ secrets.XXX ｝｝` 注入。
- **GitLab CI**: 使用 `trigger` 触发下游流水线，`CI_JOB_TOKEN` 用于内部认证。
- **Jenkins**: 使用 `input` 步骤实现手动触发，`withCredentials` 管理令牌。
- **其他平台**: 参照各平台文档进行语法映射。

---

## 注意事项

1. 发布流水线应设置严格的权限控制，仅允许授权用户手动触发。
2. 产物签名和来源证明有助于确保供应链安全。
3. 版本 PR 机制允许团队在发布前审核变更日志和版本号。
4. 发布后处理可根据实际需求裁剪（如不需要通知下游则可移除）。
5. 建议配置试运行模式，用于验证发布流程而不实际执行发布操作。
