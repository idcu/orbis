# B-F04 多仓库同步模板（完整版）

> **模板 ID**: B-F04
> **模板名称**: 多仓库同步模板（完整版）
> **版本**: 1.0.0
> **适用场景**: 完整多仓库同步方案，支持双向同步、冲突处理、Webhook 集成及多种触发方式

---

## 概述

本模板提供一个完整的多仓库同步方案，支持：

- **双向同步** - 在两个仓库之间双向同步代码
- **冲突处理** - 自动检测并处理同步冲突
- **多种触发方式** - 定时同步、手动触发、Webhook 触发
- **SSH 密钥配置** - 安全的认证方式
- **同步日志** - 完整的同步记录和通知

---

## 流水线配置

```yaml
# ｛｛项目名称｝｝ 多仓库同步（完整版）
# 适用于 CI 平台

# ====== 全局变量 ======
variables:
  SOURCE_REPO: "｛｛源仓库地址｝｝"
  TARGET_REPO: "｛｛目标仓库地址｝｝"
  SYNC_BRANCHES: "｛｛同步分支列表｝｝"
  CONFLICT_STRATEGY: "｛｛冲突处理策略｝｝"

# ====== 并发控制 ======
concurrency:
  group: "repo-sync"
  cancel-in-progress: false  # 同步不允许取消

# ====== 触发条件 ======
triggers:
  schedule:
    - cron: "｛｛定时表达式｝｝"
  manual:
    inputs:
      direction:
        description: "同步方向"
        type: "choice"
        options: ["source-to-target", "target-to-source", "bidirectional"]
        default: "bidirectional"
      force:
        description: "强制同步（覆盖冲突）"
        type: "boolean"
        default: false
  webhook:
    events:
      - "push"

# ====== 作业 1: 配置 SSH 密钥 ======
jobs:
  setup:
    name: "配置同步环境"
    runs-on: "｛｛运行环境｝｝"
    steps:
      - name: 配置 SSH 密钥
        run: |
          mkdir -p ~/.ssh
          echo "$｛｛SSH私钥｝｝" > ~/.ssh/sync_key
          chmod 600 ~/.ssh/sync_key
          ssh-keyscan ｛｛源仓库域名｝｝ >> ~/.ssh/known_hosts
          ssh-keyscan ｛｛目标仓库域名｝｝ >> ~/.ssh/known_hosts

      - name: 配置 Git
        run: |
          git config --global user.name "｛｛同步机器人用户名｝｝"
          git config --global user.email "｛｛同步机器人邮箱｝｝"

      - name: 验证连接
        run: |
          git ls-remote ｛｛源仓库SSH地址｝｝ > /dev/null 2>&1 && echo "源仓库连接成功" || echo "源仓库连接失败"
          git ls-remote ｛｛目标仓库SSH地址｝｝ > /dev/null 2>&1 && echo "目标仓库连接成功" || echo "目标仓库连接失败"

  # ====== 作业 2: 检测变更 ======
  detect-changes:
    name: "检测变更"
    runs-on: "｛｛运行环境｝｝"
    needs: [setup]
    steps:
      - name: 克隆源仓库
        run: |
          git clone ｛｛源仓库SSH地址｝｝ source-repo

      - name: 克隆目标仓库
        run: |
          git clone ｛｛目标仓库SSH地址｝｝ target-repo

      - name: 比较差异
        id: diff
        run: |
          CHANGED_BRANCHES=""
          for BRANCH in ｛｛同步分支列表｝｝; do
            cd source-repo && git checkout "$BRANCH" 2>/dev/null
            cd ../target-repo && git checkout "$BRANCH" 2>/dev/null
            DIFF=$(git diff source-repo/"$BRANCH" target-repo/"$BRANCH" --stat)
            if [ -n "$DIFF" ]; then
              echo "分支 $BRANCH 有差异:"
              echo "$DIFF"
              CHANGED_BRANCHES="$CHANGED_BRANCHES $BRANCH"
            fi
          done
          echo "changed=$CHANGED_BRANCHES" >> ｛｛步骤输出变量｝｝

      - name: 输出变更摘要
        run: |
          if [ -n "$｛｛步骤输出.变更分支｝｝" ]; then
            echo "需要同步的分支: $｛｛步骤输出.变更分支｝｝"
          else
            echo "所有分支已是最新，无需同步"
          fi

  # ====== 作业 3: 执行同步 ======
  sync:
    name: "执行仓库同步"
    runs-on: "｛｛运行环境｝｝"
    needs: [detect-changes]
    if: "$｛｛步骤输出.变更分支 != ''｝｝"
    steps:
      - name: 克隆两个仓库
        run: |
          git clone ｛｛源仓库SSH地址｝｝ source-repo
          git clone ｛｛目标仓库SSH地址｝｝ target-repo

      - name: 同步: 源 → 目标
        run: |
          for BRANCH in ｛｛同步分支列表｝｝; do
            cd source-repo
            git checkout "$BRANCH"

            cd ../target-repo
            git checkout "$BRANCH" || git checkout -b "$BRANCH"

            # 尝试合并
            if git merge "source-repo/$BRANCH" --no-edit 2>/dev/null; then
              echo "分支 $BRANCH 同步成功（源→目标）"
              git push origin "$BRANCH"
            else
              echo "分支 $BRANCH 存在冲突"
              case "｛｛冲突处理策略｝｝" in
                source-wins)
                  echo "策略: 源仓库优先"
                  git merge "source-repo/$BRANCH" -X theirs --no-edit
                  git push origin "$BRANCH"
                  ;;
                target-wins)
                  echo "策略: 目标仓库优先"
                  git merge "source-repo/$BRANCH" -X ours --no-edit
                  git push origin "$BRANCH"
                  ;;
                manual)
                  echo "策略: 需要人工处理"
                  echo "CONFLICT_BRANCH=$BRANCH" >> ｛｛环境变量文件｝｝
                  ;;
              fi
            fi
          done

      - name: 同步: 目标 → 源（双向模式）
        if: "$｛｛同步方向 == bidirectional || 同步方向 == target-to-source｝｝"
        run: |
          for BRANCH in ｛｛同步分支列表｝｝; do
            cd target-repo
            git checkout "$BRANCH"

            cd ../source-repo
            git checkout "$BRANCH" || git checkout -b "$BRANCH"

            if git merge "target-repo/$BRANCH" --no-edit 2>/dev/null; then
              echo "分支 $BRANCH 同步成功（目标→源）"
              git push origin "$BRANCH"
            else
              echo "分支 $BRANCH 存在冲突（目标→源），跳过"
            fi
          done

      - name: 同步标签
        run: |
          # 同步所有标签
          cd source-repo
          git push ｛｛目标仓库SSH地址｝｝ --tags 2>/dev/null || true
          cd ../target-repo
          git push ｛｛源仓库SSH地址｝｝ --tags 2>/dev/null || true

  # ====== 作业 4: 同步通知 ======
  notify:
    name: "同步结果通知"
    runs-on: "｛｛运行环境｝｝"
    needs: [sync]
    if: "$｛｛always()｝｝"
    steps:
      - name: 发送同步报告
        run: |
          ｛｛发送通知命令｝｝ \
            --title "仓库同步报告" \
            --body "同步完成。方向: $｛｛同步方向｝｝，状态: $｛｛同步状态｝｝" \
            --webhook "｛｛通知地址｝｝"
```

## 冲突处理策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| `source-wins` | 源仓库的变更优先覆盖 | 源仓库为主仓库 |
| `target-wins` | 目标仓库的变更优先保留 | 目标仓库为主仓库 |
| `manual` | 标记冲突，等待人工处理 | 关键项目，需人工审核 |

## 同步流程

```
定时/手动/Webhook 触发
        │
        ▼
  配置 SSH 密钥 ──→ 验证连接
        │
        ▼
  检测变更 ──→ 无变更 → 结束
        │
     有变更
        │
        ▼
  执行同步（源→目标）
        │
        ▼
  执行同步（目标→源）[双向模式]
        │
        ▼
  同步标签
        │
        ▼
  发送同步报告
```

## Webhook 集成

在源仓库和目标仓库分别配置 Webhook，当有 push 事件时自动触发同步流水线：

```json
{
  "url": "｛｛CI平台Webhook地址｝｝",
  "content_type": "json",
  "secret": "｛｛Webhook密钥｝｝",
  "events": ["push"]
}
```

## 适配指南

- **GitHub Actions**: 使用 `repository_dispatch` 触发，`secrets` 管理密钥。
- **GitLab CI**: 使用 `triggers` + `pipeline` 触发，`CI/CD Variables` 管理密钥。
- **Jenkins**: 使用 `Generic Webhook Trigger` 插件，`credentials` 管理密钥。
- **其他平台**: 参照各平台的 Webhook 和密钥管理机制。

---

## 注意事项

1. 双向同步存在冲突风险，建议明确主仓库并使用单向同步。
2. SSH 密钥应使用专用的部署密钥（Deploy Key），仅授予最小权限。
3. 定时同步的频率应根据团队协作节奏设定，避免过于频繁。
4. 建议保留同步日志，便于排查同步问题。
5. Webhook 触发需注意防止无限循环（A 推送触发同步到 B，B 推送又触发同步到 A）。
